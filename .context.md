# tools — Developer Context

This file is excluded from Cloudflare deployment. It is for developer reference only.

## Project Overview

A collection of browser-based utility tools served as static assets via Cloudflare Workers. Each tool is a **single-file** `index.html` with no build step, no npm, no bundler. Shared logic lives in `utils/`.

Deployed at: `https://tools.bluejar.org/`
GitHub: `https://github.com/humor4fun/tools`
License: AGPL-3.0

---

## How to Add a New Tool

1. Create `tools/<toolname>/index.html` (single file, no dependencies outside `utils/`)
2. Add a tile in `tools/index.html` (follow existing tile pattern)
3. Add a row in `tools/README.md`
4. Load shared utils (see canonical load order below)
5. Call `renderHeader({title, toolIconSvg, githubPath})` in init
6. Call `initSidebarSections(storageKey, sectionIds[])` if the tool has a sidebar
7. Write a readme file in `tools/<toolname>/README.md` describing the tool, features, design, and usage

---

## How Shared Utils Work

All utils are plain JS globals loaded via `<script src="../utils/x.js">`. No modules, no bundler — works from `file://` and Cloudflare Workers alike.

CSS utils self-load their companion `.css` file on first script execution.

Always attempt to re-use components from the `utils/` folder. If any of these modules are edited, all tools must be evaluated for impact.

See `utils/README.md` for full API docs.

## Flexible Config Pattern for All Utilities

All utilities expose a **flexible configuration system** with sensible defaults. Tools can use utilities with zero config (relying on defaults) or override any option as needed. This pattern ensures consistency, discoverability, and forward compatibility.

### Pattern Structure

Each utility follows this standard:

**1. Default Config Object** (at top of utility, after `'use strict'`)
```js
const DEFAULT_CONFIG = {
  // Sensible defaults extracted from current tool usage
  key1: value1,
  key2: value2,
  // ... etc
};
```

**2. Public Initialization Function** (accepts optional config override)
```js
window.initUtility = function(userConfig) {
  const config = { ...DEFAULT_CONFIG, ...userConfig };
  // ... initialize with merged config
  return controllerOrNoop;
};
```

Or for class-based utilities:
```js
class MyUtility {
  constructor(elementId, userConfig = {}) {
    const config = { ...DEFAULT_CONFIG, ...userConfig };
    // ... initialize
  }
}
```

**3. Config Merging** (shallow merge, top-level keys only)
```js
const config = { ...DEFAULT_CONFIG, ...userConfig };
```

**4. Error Handling** — Three-tier reporting (if available)
- **Tier 1**: Console (always)
  ```js
  console.error('ActivityLog:', msg);
  ```
- **Tier 2**: Toast (if `window.toast` available)
  ```js
  if (typeof window.toast === 'function') {
    window.toast(msg, 'warn', 3000);
  }
  ```
- **Tier 3**: Activity Log (if `window._activityLog` available)
  ```js
  if (window._activityLog && typeof window._activityLog.error === 'function') {
    window._activityLog.error(msg);
  }
  ```

### Key Benefits

| Benefit | Explanation |
|---------|---|
| **Zero-config usage** | Tools call `initUtility()` with no arguments; defaults apply automatically |
| **Customizable** | Tools override only what they need, inherit rest from defaults |
| **Discoverable** | Default config serves as inline documentation |
| **Consistent** | All utils follow the same pattern; developers know what to expect |
| **Forward-compatible** | New config options don't break existing code (old calls still work) |
| **Backward-compatible** | Old function signatures remain unchanged; config API added alongside |

### Usage Examples

**Using defaults (minimal code):**
```js
const log = new ActivityLog('activity-log');
// All DEFAULT_CONFIG values apply
```

**Partial overrides (customize what matters):**
```js
const stats = new Stats(editor, { 
  updateOnInput: true,
  debounceMs: 200  // override debounce, keep other defaults
});
```

**Factory function pattern (for non-class utilities):**
```js
const progress = window.initProgress({
  containerId: 'my-progress',
  label: false
});
// Returns a controller object with methods: .setValue(), .setError(), etc.
```

### Naming Convention

- **camelCase** for all config keys (e.g., `storageKey`, `updateOnInput`, `maxEntries`)
- **Boolean flags** prefix with `enable`, `show`, `allow`, `persist` (e.g., `enableLogging`, `showTimestamps`)
- **Numeric limits** use `max`, `min`, `default` (e.g., `maxEntries`, `defaultDuration`)
- **Callback functions** prefix with `on` (e.g., `onToggle`, `onError`, `onSave`)

## Design System

Always use `tokens.css`. Never hard-code colors or font stacks that are already in tokens.

Key tokens:
- `--bg`, `--bg-sidebar`, `--bg-hover`, `--bg-active` — backgrounds
- `--border`, `--border-light` — borders
- `--text`, `--text-muted`, `--text-dim`, `--text-heading` — text
- `--accent-blue`, `--accent-green` — accents; `--accent` = alias for `--accent-blue`
- `--radius` / `--r` — border radius (6px)
- `--font-mono`, `--font-sans` / `--font-ui` — typefaces

Shared structural pattern:
- `#app` CSS grid: `header (full width) | sidebar (290px) + main (1fr) | statusbar (full width)`
- `header.js` generates: `#hdr-logo`, `.hdr-sep`, `#hdr-title`, `#tab-bar`, `#header-actions`, `#hdr-gh`
- Sidebar: `<div class="sb-sec" id="sec-foo"><div class="sb-hdr">…<span class="sb-chevron">…</span></div><div class="sb-body">…</div></div>`
- Status bar: `<div id="stbar">` with `<span id="stats">` or similar

## Coding Conventions

- Vanilla JS only — no frameworks, no npm, no build step
- `'use strict'` in every `<script>`
- CSS custom properties (tokens) for all colors/spacing
- localStorage keys: `<toolname>_prefs` for general prefs, `<toolname>_tabs` for tab state, `<toolname>_prefs_collapsed` for sidebar collapse (managed by sidebar.js)
- Git history uses descriptive commit messages matching the change scope

## How to Add a New Util

1. Create `utils/<name>.js` (and `utils/<name>.css` if needed)
2. Expose a single `window.*` global
3. If it has CSS, self-load it from `document.currentScript.src`
4. Document in `utils/README.md`
5. Add to the canonical load order above
6. Add to `wrangler.jsonc` `assets.exclude` if it's a non-deployable doc file
